# iGaming 智能客服系统 — 说明文档（Lark + Telegram × Intercom Fin）

> 目标：客户在 **Lark / Telegram** 上发消息，就像和真人客服聊天。系统会把消息变成 **Intercom 里的一条真实对话**，由 **Fin AI** 自动回答（可查实时的存款/取款状态），需要时 **人工客服可在同一条对话里接手**，回复直接发回客户的 Lark/Telegram。所有渠道汇总到 **一个 Intercom 收件箱** 统一管理。
>
> 本文档面向老板与技术同事，力求"看完就懂、无需录视频"。

---

## 一、一句话说明它做什么

```
客户(Lark / Telegram) ──> 我们的后端 ──> Intercom 一条真实对话
                                           ├─ Fin AI 自动回答（含实时数据查询）
                                           └─ 人工客服在同一对话里接手
                          回复 ──────────> 自动发回客户的 Lark / Telegram
```

- **统一**：Lark、Telegram 两个渠道，进同一个 Intercom 收件箱。
- **智能**：Fin AI 自动回答常见问题，并能查"存款到账没""提款到哪了"等实时状态。
- **可接手**：客服随时在同一条对话里打字，客户在 Lark/Telegram 收到。
- **安全**：实时数据会脱敏（如银行卡只显示后四位）、用安全话术，不泄露内部代码。
- **无服务器**：后端跑在 Cloudflare（按量、免运维服务器）。

## 二、整体架构（一张图看懂）

```
            ┌──────────────┐         ┌──────────────┐
  客户  ──> │   Lark App   │         │  Telegram    │  <── 客户
            └──────┬───────┘         └──────┬───────┘
                   │  (消息事件 webhook)     │
                   ▼                         ▼
         ┌───────────────────────────────────────────┐
         │      Cloudflare Worker（后端大脑）          │
         │  /lark/webhook   /telegram/webhook          │  收消息
         │  /intercom/webhook                          │  收人工回复
         │  /api/deposit  /api/withdrawal  /api/kyc    │  数据接口(给Fin查)
         │                                             │
         │  Cloudflare KV（状态存储：谁对应哪条对话）  │
         └───────────────┬─────────────────▲──────────┘
                         │ 新建/复用对话    │ 轮询取 Fin 回复 / 收人工回复
                         ▼                  │
         ┌───────────────────────────────────────────┐
         │            Intercom（DKM-IGAMING）          │
         │  · 一条真实可回复的对话（不是只读的外部对话）│
         │  · Workflow：客户发首条消息 → 让 Fin 处理   │
         │  · Fin AI：回答 + 调用 Data Connector 查数据 │
         │  · 人工客服：在同一对话里接手回复            │
         └───────────────────────────────────────────┘
```

**关键点**：客户在 Lark/Telegram 发的消息，会被我们后端做成 Intercom 里一条**正常的、可回复的**对话（顶部**没有**"This conversation is handled through an external system / 无法回复"的提示）。这样 Fin 和人工都能在**同一条对话**里说话。

## 三、一条消息的完整流程（分步详解）

以客户在 Lark 问"我的存款还没到账"为例：

1. **客户发消息**：Lark 把这条消息以事件（webhook）发到我们的 `https://tg-fin-bridge.askada.workers.dev/lark/webhook`。
2. **后端接收**：Worker 先做三件事——
   - 校验 Lark 的 **Verification Token**（确认是 Lark 发来的，不是别人伪造）；
   - 用事件的 **event_id 去重**（Lark 在我们回慢时会重发，去重避免重复处理）；
   - **立刻回 200** 给 Lark，真正的处理放到后台跑（避免 Lark 因超时重发）。
3. **路由处理**（`routeInbound`）：因为"工作流桥接"开关对 Lark 打开，进入桥接流程——
   - **确保联系人**：在 Intercom 建/找该客户的联系人（`external_id = lark_<用户ID>`），并打上属性 **`source_channel = lark`**（Workflow 靠它识别这是 Lark 渠道）；
   - **建/复用对话**：如果该客户已有一条**打开的**对话，就在这条里以"**客户身份**"追加这句话；否则用 Intercom 的 **Conversations API** 新建一条**普通对话**（`from.type = user`）。
4. **Intercom 的 Workflow 触发**：规则是"**客户发出第一条消息** + 受众 `source_channel = lark/telegram`"→ 动作"**让 Fin 处理（Let Fin handle）**"。于是 Fin 在**这条对话里**回答。
   - 如果客户问的是"存款/提款状态"，Fin 会调用我们配的 **Data Connector**（见第四节 5、6），去 `/api/deposit`、`/api/withdrawal` 查实时数据，再用结果回答。
5. **把 Fin 的回答发回 Lark**：Fin 的回复在 Intercom 里产生（它是"机器人"身份）。我们后端用 **轮询（poll）** 盯着这条对话约 30 秒，一有 Fin 的新回复就发回客户的 Lark。
6. **人工接手（可选）**：当客户要求人工、或客服想介入时，客服直接在**这条 Intercom 对话**里打字回复。Intercom 发 `conversation.admin.replied` 事件到 `/intercom/webhook`，后端把人工回复发回 Lark（前面加 👤 区分人工）。
7. **结束**：对话可由客服关闭，或客户发 `/reset` 重置；下次再发消息会开一条新对话、重新触发 Workflow。

> Telegram 流程完全一样，只是入口是 `/telegram/webhook`，渠道是 `telegram`。

## 四、各组成部分说明

### 1. Lark / Telegram（客户渠道）
- **Lark**：一个 Lark"自定义应用（机器人）"，名字/头像可设成"Ask Ada"。客户私聊这个机器人。
- **Telegram**：一个 Telegram 机器人（BotFather 创建）。
- 它们只负责"收客户消息、发回客户消息"，不含业务逻辑。

### 2. Cloudflare Worker（后端大脑）
- 一段无服务器代码，是整套系统的中枢。负责：接收两个渠道的消息、把消息桥接到 Intercom、把回复发回渠道、以及对外提供数据接口给 Fin 调用。
- 地址：`https://tg-fin-bridge.askada.workers.dev`
- 主要入口（路由）：

| 路由 | 作用 |
|---|---|
| `POST /lark/webhook` | 接收 Lark 客户消息 |
| `POST /telegram/webhook` | 接收 Telegram 客户消息 |
| `POST /intercom/webhook` | 接收 Intercom 人工客服的回复/关闭事件 |
| `GET /api/deposit` `…/withdrawal` `…/kyc` | 给 Fin 调用的实时数据接口 |
| `POST /fin/webhook` | （旧方案保留）Fin Agent API 回调，当前桥接模式不使用 |
| `GET /` | 健康检查，返回 `tg-fin-bridge: ok` |

### 3. Cloudflare KV（状态存储）
- 一个键值数据库，存"谁对应哪条对话"等映射关系。命名空间：`tg_fin_state`。
- 主要的键（见第六节详解）：`wf:`、`icid:`、`contact:`、`evt:`、`relayedpart:` 等。

### 4. Intercom Fin + Workflow（AI 自动回答）
- **Workspace**：DKM-IGAMING（美国区，`api.intercom.io`）。
- **Workflow（工作流）**：触发条件"客户发出第一条消息"，渠道勾 **Web**，受众 **`source_channel` 是 `lark` 或 `telegram`**，动作"**让 Fin 处理**（用你的知识库内容回答；不确定时向客户追问；可升级人工）"。
- Fin 在对话里多轮回答，需要时调用下面的数据接口。

### 5. & 6. Intercom Data Connector（数据接口：存款 / 提款）
- 在 Intercom 里配置的"自定义动作"，让 Fin 能查实时数据：

| 动作 | 请求 | 输入（Fin 向客户索取） |
|---|---|---|
| 查存款状态 | `GET /api/deposit?member_id=&deposit_id=` | 会员ID、存款单号 |
| 查提款状态 | `GET /api/withdrawal?member_id=&withdrawal_id=` | 会员ID、提款单号 |

- 两个动作都带认证头 `Authorization: Bearer <GATEWAY_API_TOKEN>`（密钥，只有 Fin 能调我们的接口）。

### 7. 数据网关（Worker 的 `/api/*`）
- 这是我们后端里"安全地暴露数据给 Fin"的一层。当前为 **演示用 mock 数据**（上线时把这层换成真实 iGaming 系统接口即可，对 Fin 的接口契约不变）。
- 三大职责（合规）：
  1. **鉴权**：没有正确的 Bearer 令牌一律 401。
  2. **脱敏**：如银行卡号只返回后四位 `****8821`。
  3. **安全话术**：把内部状态/风控代码翻译成"对客户安全的说法"，绝不返回内部代码或风险分。

## 五、关键设计（为什么这样做）

1. **为什么要把消息"变成" Intercom 对话？**
   Intercom 没有 Lark/Telegram 的"原生渠道"。如果用 Fin Agent API，会生成一条**只读**的"外部系统"对话，客服无法在 Intercom 里回复。我们改用 **Conversations API** 新建一条**普通对话**，它可回复、无只读提示，Fin 和人工都能在里面说话。这就是老板要的"一个对话里 AI 和人工一起"。

2. **为什么一个客户复用一条对话？**
   同一客户的多轮对话都放在**同一条打开的对话**里，方便客服看上下文。只有对话被关闭、或客户 `/reset` 时，下条消息才开新对话（并重新触发 Workflow 让 Fin 再次接手）。

3. **为什么用"轮询"把 Fin 的回复发回渠道？**
   Fin 是"机器人(bot)"身份，它的回复**不会**触发 Intercom 的 `conversation.admin.replied` webhook（那个只对**真人**客服触发）。所以 Fin 的回复我们用**主动轮询对话**来获取；**人工**回复则用 webhook。两条路都做了**去重**，不会重复发。

4. **去重与防回环**
   - Lark 事件按 `event_id` 去重（防重发）。
   - 已转发的对话消息按 `part id` 去重（防重复转发）。
   - 只转发 Fin(bot) 与人工(admin) 的"comment"，不转发客户自己的消息（防回环），也不转发内部备注(note)。

5. **多语言**
   后端原样转发 Fin 产出的文字。让 Fin 用客户语言回答（如中文问→中文答）是 **Intercom 端的 Fin 语言设置**（开启对应语言 + 自动翻译知识库内容）。

## 六、状态存储（KV 键的含义）

在 Cloudflare → Storage & databases → **Workers KV** → `tg-fin-bridge-TG_FIN_STATE` 里可看到：

| 键（key） | 值 | 含义 |
|---|---|---|
| `wf:{渠道}:{用户ID}` | 对话信息 | 该客户当前对应的 Intercom 对话（正向映射） |
| `icid:{对话ID}` | `{channel, cuid}` | 由 Intercom 对话反查是哪个渠道的哪个客户（转发回去用） |
| `contact:{渠道}:{用户ID}` | 联系人ID | 该客户在 Intercom 的联系人 |
| `evt:{事件ID}` | 标记 | Lark 事件去重（24h 过期） |
| `relayedpart:{partID}` | 标记 | 已转发的对话消息去重（24h 过期） |
| `handoff:` / `conv:` / `transcript:` 等 | — | 旧方案/兼容用的状态 |

> 例：`icid:215474811570006` 的值是 `{"channel":"telegram","cuid":"8475547668"}` —— 说明这条 Intercom 对话属于 Telegram 用户 8475547668。

## 八、配置清单（系统是怎么搭起来的）

**A. Lark 自定义应用**（open.larksuite.com → 开发者后台）
- 启用 **机器人(Bot)**。
- 权限(Scopes)：`im:message`、`im:message:send_as_bot`、**`im:message.p2p_msg`（接收单聊消息——必加，否则私聊不触发事件）**。
- 事件订阅：请求地址 `https://tg-fin-bridge.askada.workers.dev/lark/webhook`，订阅 `im.message.receive_v1`；**不启用 Encrypt Key**；记下 **Verification Token**。
- 发布/上线应用。
- 把 **App ID / App Secret / Verification Token** 配到 Worker 的 secrets（`LARK_APP_ID`/`LARK_APP_SECRET`/`LARK_VERIFICATION_TOKEN`）。

**B. Intercom（DKM-IGAMING）**
- 新建联系人文本属性 **`source_channel`**。
- 新建 **Workflow**：触发"客户发出第一条消息"，渠道 Web，受众 `source_channel 是 lark/telegram`，动作"让 Fin 处理"。
- 新建两个 **Data Connector / 自定义动作**：查存款、查提款（见第四节）。
- （多语言）在 Fin 语言设置里开启中文 + 自动翻译。

**C. Cloudflare**
- Worker 已部署；KV 命名空间 `tg_fin_state` 已绑定；secrets 已配置；`INTERCOM_WORKFLOW_BRIDGE = "lark,telegram"`。

## 十、已解决的常见问题（踩过的坑）

- **私聊机器人没反应**：Lark 缺 `im:message.p2p_msg`（接收单聊消息）权限——已加。
- **/reset 触发两次**：Lark 回应慢导致重发——已用 event_id 去重 + 立即回 200 后台处理。
- **Fin 回答没发到渠道**：Fin"先说稍等→查询→再正式回答"耗时较长，早期轮询窗口太短——已改为最长约 30 秒、且"Fin 还没开口就不提前结束"。
- **链接变纯文本**：轮询取对话时要取 **HTML** 正文，才能把 `<a>` 链接转成"文字(网址)"，Lark/Telegram 自动识别为可点链接——已修。
- **总是自动关闭对话、变成多条**：Fin 回答后会"自动解决/关闭"。已改为"对话开着就复用同一条"；如需对话长时间保持打开，可在 Fin 设置里调高/关闭"自动关闭闲置对话"。
- **中文问、英文答**：这是 Intercom 端 Fin 的语言设置，需开启对应语言 + 自动翻译（见第八节 B）。
